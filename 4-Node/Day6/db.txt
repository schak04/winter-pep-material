//it gives 3 parts of data

(a) -->| 4 bytes |     5 bytes      | 3 bytes |
|timestamp| random/machine   | counter |

------>
699bd9bd | af9c140eaf | df0404
^^^^^^^^   ^^^^^^^^^^   ^^^^^^
timestamp     random      counter

//1. Timestamp (first 8 hex chars)

   699bd9bd

Represents the creation time of the document.
Stored as Unix time (seconds since Jan 1, 1970).
This is why ObjectIds are roughly sortable by creation date.

//2.Random / Machine Value (next 10 hex chars)
af9c140eaf

Older MongoDB versions used:
machine id + process id
Modern MongoDB drivers use:
a random 5-byte value

Purpose:-
Ensures IDs generated on different machines/processes don’t collide.
Makes ObjectIds globally unique without a central server.

//3.Counter (last 6 hex chars)
df0404

A 3-byte incrementing number.
Starts at a random value and increases for each new ObjectId created by that process.

Purpose:-
Prevents duplicates when many documents are created in the same second.

Note:->ObjectId =
[ TIME ] + [ RANDOM SOURCE ] + [ INCREMENT ]

// Why not index everything?
Because indexes also have costs, and too many indexes can actually make your database slower.
//Why not index everything in a book?
Because an index is supposed to help you find important things faster — not list every single word or detail.
If you indexed everything in a book, the index would become useless 😄

//Disadvantages of Indexing
Indexes are stored separately and take disk space.
More indexes = more storage usage.
Slower Write Operations (Insert/Update/Delete)
Every time you insert a user, MongoDB must also update the index.
Over-Indexing Hurts Performance
Many beginners create indexes on every field.
Maintenance Overhead
Indexes must stay updated when data changes.
