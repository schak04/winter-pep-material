**Devops:-

Q.What is DevOps?

DevOps is a culture, philosophy, and set of practices that combines Development (Dev) and
Operations (Ops) teams to improve collaboration, automate workflows, and deliver software faster and more reliably.

Traditionally, development and operations teams worked separately, which caused delays, communication gaps, and deployment failures.
DevOps removes this gap by encouraging shared responsibility, automation, and continuous improvement.

Goal of DevOps:

* Faster delivery
* Better quality
* Continuous integration and deployment
* Improved collaboration

DevOps focuses on automation, monitoring, continuous feedback, and rapid releases.
------------------------------------------------------------------------------
## 5. Dev vs Ops Gap

### Development Team (Dev)

* Focus: Writing code
* Goal: Deliver new features quickly
* Works in fast cycles

### Operations Team (Ops)

* Focus: Stability and reliability
* Goal: Keep system running without failure
* Prefers fewer changes

Problem:
Dev wants speed, Ops wants stability → Conflict occurs.
DevOps removes this gap by encouraging collaboration, shared responsibility, and automation.
---------------------------------------------------------------------------------

## 6. DevOps Lifecycle
DevOps is a continuous cycle where development and operations work together to deliver software faster and more reliably.

The DevOps lifecycle is continuous and includes:
1. Plan
2. Develop
3. Build
4. Test
5. Release
6. Deploy
7. Operate
8. Monitor

1.Plan:-
Define project requirements and goals
Create user stories and backlog
Decide timeline and resources

2.Develop:-
Write source code
Use version control systems (like Git)
Collaborate with team members

3.Build:-

Compile the source code
Convert code into executable files
Generate build artifacts

4.Test:-

Perform automated and manual testing
Detect and fix bugs
Ensure software quality

5.Release:-

Prepare final build for deployment
Verify release readiness
Get approval if required

5.Deploy:-

Deploy application to production or staging
Use automated deployment tools (CI/CD)
Make software available to users

6.Operate:-

Manage and maintain the live application
Ensure performance, stability, and security
Handle infrastructure and servers

7.Monitor:-

Track performance and errors
Collect logs and metrics
Gather feedback for improvement

This cycle repeats continuously for continuous improvement.

Note:-
Common DevOps Tools:
• Git – Version Control
• Docker – Containerization
• Jenkins – CI/CD Automation
• Kubernetes – Container Orchestration
• Terraform – Infrastructure as Code
-------------------------------------------------------------------------------------------------

## 7. CI/CD Meaning

CI/CD stands for Continuous Integration and Continuous Delivery (or Deployment).
It is a DevOps practice that automates building, testing, and deploying applications.
-------------------------------------------------------------------------------------------------

## 8. CI vs CD vs Continuous Deployment

### Continuous Integration (CI)

* Developers frequently merge code into a shared repository
* Automated build and testing
* Detect bugs early

### Continuous Delivery (CD)

* Code is automatically tested and prepared for release
* Deployment to production requires manual approval

### What is Continuous Deployment?

Continuous Deployment automatically releases code to production without manual approval.

### CI vs CD vs Continuous Deployment

| Feature                 | CI           | Continuous Delivery | Continuous Deployment |
| ----------------------- | ------------ | ------------------- | --------------------- |
| Code Integration        | Yes          | Yes                 | Yes                   |
| Automated Testing       | Yes          | Yes                 | Yes                   |
| Manual Approval         | Not Required | Required            | Not Required          |
| Auto Production Release | No           | No                  | Yes                   |

**Cloud:-

Q. Why Cloud?

Cloud computing allows users to access computing resources (servers, storage, databases) over the internet instead of owning physical hardware.

Reasons to use Cloud:-

1. Cost-effective (Pay-as-you-go)
2. Scalable
3. High availability
4. Remote access
5. Faster deployment
6. Reduced maintenance
-------------------------------------------------------------------------------------------

## On-Premises vs Cloud

### On-Premises

* Infrastructure owned and managed by company
* High initial investment
* Limited scalability
* Requires maintenance team

### Cloud

* Infrastructure provided by cloud provider
* Pay-as-you-go model
* Easily scalable
* Managed by provider

### Comparison

| Feature       | On-Prem         | Cloud            |
| ------------- | --------------- | ---------------- |
| Cost          | High initial    | Pay-as-you-go    |
| Scalability   | Limited         | High             |
| Maintenance   | Company managed | Provider managed |
| Accessibility | Limited         | Global access    |

-----------------------------------------------------------------------------------------

## IaaS, PaaS, SaaS Models

### IaaS (Infrastructure as a Service)

* Provides virtual machines, storage, networks
* User manages OS and applications
* Example: Virtual servers

### PaaS (Platform as a Service)

* Provides platform to develop and deploy applications
* User manages application only
* Provider manages infrastructure

### SaaS (Software as a Service)

* Ready-to-use software over internet
* No installation required
* Example: Email services, Google Docs

### Service Model Comparison

| Model | What You Manage  | Provider Manages |
| ----- | ---------------- | ---------------- |
| IaaS  | Applications, OS | Hardware         |
| PaaS  | Applications     | OS + Hardware    |
| SaaS  | Nothing          | Everything       |

-----------------------------------------------------------------------

## Public vs Private vs Hybrid Cloud

### Public Cloud

* Shared infrastructure
* Available to general public
* Cost-effective

### Private Cloud

* Dedicated infrastructure for one organization
* Higher security
* More control

### Hybrid Cloud

* Combination of public and private cloud
* Flexible and balanced approach

### Deployment Model Comparison

| Type    | Ownership    | Security | Cost     |
| ------- | ------------ | -------- | -------- |
| Public  | Provider     | Medium   | Low      |
| Private | Organization | High     | High     |
| Hybrid  | Both         | Balanced | Moderate |
-------------------------------------------------------------------------------------

• You build an application → Development
• You automate testing & deployment → DevOps
• You host it on AWS or Azure → Cloud Computing

Q. How DevOps and Cloud Work Together
In modern IT companies, DevOps and Cloud are closely connected. Cloud platforms provide infrastructure, and DevOps practices automate deployment on that infrastructure.
Example Workflow:
1. Developer writes code
2. Code pushed to Git repository
3. CI/CD pipeline builds and tests the application
4. Docker creates a container image
5. Application deployed to cloud server
